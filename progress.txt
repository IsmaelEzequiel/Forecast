## Codebase Patterns
- Mix project root is at `weather_edge/` subdirectory (not repo root)
- Run `mix compile --warnings-as-errors` from `weather_edge/` for typecheck
- Oban cron workers are defined in `config/config.exs` - worker modules must exist before compilation
- Test config uses `Oban, testing: :inline` to disable async job processing
- Polymarket credentials are in runtime.exs under `:polymarket` key
- Trading safety config is under `:trading` key, forecasts under `:forecasts` key
- Docker: `docker-compose up` from `weather_edge/` dir; app runs migrations on startup via entrypoint.sh
- Production release uses `WeatherEdge.Release.migrate/0` for Ecto migrations
- Schemas follow `lib/weather_edge/<context>/<schema>.ex` pattern (e.g., `stations/station.ex`)
- Migrations use `timestamps(type: :timestamptz)`, schemas use `timestamps(type: :utc_datetime)`
- Hypertable migrations use `up/down` (not `change`), composite PK via ALTER TABLE, and `SELECT create_hypertable()`
- Trading context lives in `lib/weather_edge/trading/` (Position, Order, OrderManager, ClobClient, DataClient, Auth)
- OrderManager uses `:persistent_term` for rate limiting (key: `{:order_manager_rate_limit, station_code}`)
- PubSub topic for trading events: `"trading:STATION_CODE"` with events `:order_placed`, `:order_filled`, `:order_failed`
- Signals context lives in `lib/weather_edge/signals/` (Signal schema)
- Calibration context lives in `lib/weather_edge/calibration/` (Accuracy schema)
- Git working directory is `weather_edge/` - use relative paths for git add/commit
- All PubSub usage goes through `WeatherEdge.PubSubHelper` - never use raw `Phoenix.PubSub` calls directly
- Probability context lives in `lib/weather_edge/probability/` (Gaussian module)
- Workers live in `lib/weather_edge/workers/` - use `use Oban.Worker, queue: :queue_name`
- When referencing a worker that doesn't exist yet, create a stub so compilation passes
- LiveView modules live in `lib/weather_edge_web/live/` - use `use WeatherEdgeWeb, :live_view`
- Router uses `live "/path", ModuleLive` for LiveView routes (not `get`)
- JS hooks are defined in `assets/js/app.js` in the `Hooks` object, passed to LiveSocket via `hooks: Hooks`
- App layout is in `lib/weather_edge_web/components/layouts/app.html.heex`
- CoreComponents defines `header/1` - avoid name collisions by prefixing custom component functions (e.g., `dashboard_header/1`)
- Custom components go in `lib/weather_edge_web/components/<name>_component.ex` under `WeatherEdgeWeb.Components.*`
- Tests go in `test/weather_edge/<context>/` mirroring `lib/` structure; use `DataCase` for DB-backed tests, `ConnCase` for web tests
- MarketCluster outcomes field is `{:array, :map}` type - use list of maps, not a single map
- OrderManager uses configurable client modules via `Application.get_env(:weather_edge, :data_client)` and `:clob_client` - mock in test config
- Mock clients for trading tests are in `test/support/mock_clients.ex` using `Process.put/get` for per-test configuration
- Docker `.env` file must exist for docker-compose (copy from `.env.example` with placeholder values)
- HTTP client tests use Bypass for mocking - use `Bypass.stub` (not `expect_once`) for error responses since Req retries on 429/500
- GammaClient uses `Application.get_env(:weather_edge, :gamma_api_url)` for configurable base URL (useful for test overrides)
- LiveView tests use `ConnCase` with `import Phoenix.LiveViewTest`; modal close uses `render_hook` since CoreComponents modal has JS-driven close
